01 / Problem Statement

Why This Pipeline
Needed to Exist

Modern software teams face a painful gap between development velocity and deployment confidence. Without proper automation, every release is a manual, error-prone event ‚Äî and security is an afterthought discovered after deployment.

"The goal was to eliminate human error from deployments entirely ‚Äî while catching security vulnerabilities before they ever reached production."

‚ö°

Manual Deployments

Each deployment required manual SSH into servers, running scripts by hand ‚Äî slow, inconsistent, and risky.

No Security Scanning

No Code Quality Gates

03 / Implementation

Building It Phase
by Phase

1 Azure VM Setup & Jenkins Configuration +

Two Azure VMs were provisioned ‚Äî one as the Jenkins master (build server) and one as the K3s node (deployment target). Jenkins was installed and configured with the required plugins: Docker Pipeline, SonarQube Scanner, OWASP Dependency-Check, Kubernetes CD, and GitHub integration.

# Azure VM provisioning (simplified) az vm create \ --resource-group devops-rg \ --name jenkins-server \ --image Ubuntu2204 \ --size Standard_B2s \ --admin-username azureuser \ --generate-ssh-keys # Install Jenkins on Ubuntu curl -fsSL https://pkg.jenkins.io/debian/jenkins.io-2023.key | tee \ /usr/share/keyrings/jenkins-keyring.asc > /dev/null apt-get install -y jenkins
2 Jenkinsfile Pipeline ‚Äî CI Stages +

The Jenkinsfile defines the complete CI pipeline as code, committed to the repository alongside application code. Each stage is a discrete, auditable step.

pipeline { agent any stages { stage('Checkout') { steps { git 'https://github.com/user/app' } } stage('Build & Test') { steps { sh 'mvn clean package' } } stage('SonarQube Analysis') { steps { withSonarQubeEnv('sonar-server') { sh 'mvn sonar:sonar' } } } stage('Quality Gate') { steps { waitForQualityGate abortPipeline: true } } stage('Docker Build & Push') { steps { sh 'docker build -t user/app:${BUILD_NUMBER} .' sh 'docker push user/app:${BUILD_NUMBER}' } } } }
3 Security Integration ‚Äî Trivy & OWASP +

Security scanning was embedded directly into the pipeline as non-negotiable gates. Trivy scans the built Docker image for OS and library CVEs. OWASP Dependency-Check scans project dependencies for known vulnerabilities. Both must pass before the image is pushed.

stage('Trivy Image Scan') { steps { sh """ trivy image \ --exit-code 1 \ --severity HIGH,CRITICAL \ --format table \ user/app:${BUILD_NUMBER} """ } } stage('OWASP Dependency Check') { steps { dependencyCheck additionalArguments: '--scan ./', odcInstallation: 'OWASP-DC' dependencyCheckPublisher pattern: '**/dependency-check-report.xml' } }
4 GitOps Deployment with ArgoCD on K3s +

ArgoCD was installed on the K3s cluster and connected to the Git repository containing Kubernetes manifests. Once Jenkins updates the manifest with the new image tag and pushes to Git, ArgoCD automatically detects the change and syncs the cluster. If the live state ever drifts from the Git-declared state, ArgoCD self-heals it automatically.

# ArgoCD Application manifest apiVersion: argoproj.io/v1alpha1 kind: Application metadata: name: my-app namespace: argocd spec: project: default source: repoURL: https://github.com/user/k8s-manifests targetRevision: HEAD path: manifests/ destination: server: https://kubernetes.default.svc namespace: production syncPolicy: automated: prune: true selfHeal: true
04 / Challenges

Problems Encountered
& How I Solved Them

üî¥ Problem
Trivy flagged HIGH vulnerabilities from base OS packages that were irrelevant to the running application, causing false positive pipeline failures.
üü¢ Solution
Created a .trivyignore file to suppress known false positives, and switched to a distroless base image to eliminate OS-layer CVEs entirely.
üî¥ Problem
ArgoCD sync was stuck in a degraded state after a bad deployment. Rolling back was not straightforward without understanding how Git-driven rollback works.
üü¢ Solution
Learned that rollback in GitOps means reverting the Git commit. Used git revert on the manifest repo, pushed, and ArgoCD self-healed back to the previous healthy state automatically.
üî¥ Problem
SonarQube Quality Gate was failing due to insufficient code coverage, blocking every build even during early development.
üü¢ Solution
Created a project-specific Quality Gate profile in SonarQube with adjusted coverage thresholds for the project's maturity stage, while progressively increasing them over time.
üî¥ Problem
Jenkins and DockerHub credentials were being hardcoded in pipeline scripts, creating a serious security risk in the repository.
üü¢ Solution
Migrated all credentials to Jenkins Credentials Manager and referenced them via withCredentials() blocks, ensuring secrets never appear in logs or source code.

06 / Lessons Learned

What I'd Do
Differently

üìä
Add Observability from Day One The pipeline lacks Prometheus + Grafana monitoring on the K3s cluster. Next iteration would instrument the app and cluster from the start, so you can see the impact of deployments in real-time.
üåê
Multi-Environment Strategy The current setup deploys to a single environment. A proper setup would have separate dev, staging, and production namespaces ‚Äî each with their own ArgoCD Application and promotion gates between them.
üîî
Pipeline Failure Notifications There are no Slack or email alerts when the pipeline fails at a security or quality gate. Adding webhook-based notifications would close the feedback loop much faster.
üèóÔ∏è
Infrastructure as Code The Azure VMs were provisioned manually via the CLI. Using Terraform to declare the infrastructure would make the entire setup reproducible in minutes.
üîë
Secrets Management Jenkins Credentials work, but a dedicated secrets manager like HashiCorp Vault or Azure Key Vault would be the production-grade approach ‚Äî especially for rotating credentials automatically.
